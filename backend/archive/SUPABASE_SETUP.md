# Supabase Setup Guide

Complete setup for Job Tracker with Supabase (PostgreSQL + Auth).

## Why Supabase?

- Free tier with generous limits
- Managed PostgreSQL (no server management)
- Automatic backups
- Built-in auth (we use FastAPI auth instead)
- Connection pooling
- Easy environment setup

## Setup Steps

### 1. Create Supabase Project

1. Go to https://supabase.com
2. Click "New Project"
3. Choose organization (or create one)
4. Set project details:
   - **Name:** job-tracker
   - **Database Password:** (generate strong password)
   - **Region:** Choose closest to you
5. Wait 2-3 minutes for provisioning

### 2. Get Database URL

1. Go to **Settings** → **Database**
2. Scroll to **Connection string**
3. Select **URI** tab
4. Copy the connection string (looks like):
   ```
   postgresql://postgres:[YOUR-PASSWORD]@db.abc123xyz.supabase.co:5432/postgres
   ```
5. Replace `[YOUR-PASSWORD]` with the password you set

### 3. Configure Backend

Edit `backend/.env`:

```bash
DATABASE_URL=postgresql://postgres:your-password@db.abc123xyz.supabase.co:5432/postgres
```

### 4. Create Tables

**Option A: Let SQLAlchemy create them (easiest)**

Start your FastAPI server - tables are auto-created:

```bash
cd backend
source venv/bin/activate
uvicorn app.main:app --reload
```

Check Supabase **Table Editor** - you should see:
- users
- ats_accounts
- applications
- application_updates

**Option B: Use Alembic migrations (production)**

```bash
alembic revision --autogenerate -m "Initial tables"
alembic upgrade head
```

### 5. Enable Row Level Security (RLS)

**Why RLS?**
- Defense-in-depth security
- Protects data even if backend credentials leak
- PostgreSQL-level enforcement

**Setup:**

1. Go to Supabase **SQL Editor**
2. Click **New Query**
3. Paste contents of `supabase_rls_setup.sql`
4. Click **Run**

This enables RLS on all tables and creates policies.

### 6. Get Service Role Key

Our FastAPI backend needs the **service_role** key to bypass RLS:

1. Go to **Settings** → **API**
2. Find **Project API keys**
3. Copy **service_role** key (starts with `eyJ...`)
4. **IMPORTANT:** Keep this secret - it bypasses RLS!

**Add to `.env`:**
```bash
SUPABASE_SERVICE_KEY=eyJhbG...your-service-key
```

**Note:** We don't actually use this in current setup since we connect directly to PostgreSQL. But it's useful if you want to use Supabase's client libraries.

## Connection Pooling

Supabase includes connection pooling by default:

**Direct connection (what we use):**
```
postgresql://postgres:password@db.abc123xyz.supabase.co:5432/postgres
```

**Pooled connection (for serverless):**
```
postgresql://postgres:password@db.abc123xyz.supabase.co:6543/postgres?pgbouncer=true
```

For Railway/Render deployment, use **pooled connection** (port 6543).

## Security Model

```
Frontend (React)
    ↓ HTTP + JWT
FastAPI Backend
    ↓ service_role connection (bypasses RLS)
PostgreSQL
    ↓ RLS policies (backup security)
Data
```

**Why service_role bypasses RLS:**
- FastAPI handles ALL authentication
- FastAPI filters queries by user_id
- RLS is a safety net if credentials leak
- Backend needs full access to manage data

**RLS protects against:**
- Direct database access with leaked credentials
- Backend bugs that skip user_id filtering
- Accidental admin queries
- Side-channel attacks

## Testing RLS

**Test that RLS works:**

1. Go to **SQL Editor** in Supabase
2. Run this query:
   ```sql
   SELECT * FROM applications;
   ```
3. You should see: **"new row violates row-level security policy"** (if you're not using service_role key)

**Test with service_role (our backend):**

Our FastAPI backend uses service_role credentials, so it can access all data (which is correct - it handles auth).

## Backups

Supabase free tier includes:
- Daily automated backups (7 days retention)
- Point-in-time recovery (paid plans)

**Manual backup:**
```bash
pg_dump "postgresql://postgres:password@db.abc123xyz.supabase.co:5432/postgres" > backup.sql
```

## Troubleshooting

**Error: "Could not connect to database"**
- Check DATABASE_URL in `.env`
- Verify password is correct
- Check if Supabase project is paused (free tier pauses after inactivity)

**Error: "SSL connection required"**
Add `?sslmode=require` to DATABASE_URL:
```
DATABASE_URL=postgresql://...postgres?sslmode=require
```

**Error: "Row level security policy violation"**
- Check if RLS policies are set up correctly
- Verify you're using the right credentials
- Our backend should use connection string (bypasses RLS automatically)

**Performance issues:**
- Use pooled connection (port 6543) for serverless
- Add indexes on frequently queried columns
- Check query performance in Supabase dashboard
